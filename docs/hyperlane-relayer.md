# Hyperlane Relayer Config

The `solver-cli configure` command generates a Hyperlane relayer config at `.config/hyperlane-relayer.json`.

## Prerequisites

- Contracts deployed (`make deploy`)
- `.env` populated with chain RPCs, keys, and Hyperlane contract addresses

## Install solver-cli tool
```bash
cargo install --path solver-cli
```

## How the files are generated

`hyperlane-relayer.json` is generated by `solver-cli configure`, which reads `.config/state.json` and signer keys from the environment. It includes every chain currently registered in state — nothing more.

A chain gets into `state.json` via one of two paths:

- **`solver-cli deploy`** — deploys OIF contracts and writes the chain to state automatically
- **`solver-cli chain add`** — manually registers a chain that already has contracts deployed elsewhere

So `make configure` alone is sufficient to regenerate the relayer config, as long as the chain is already in state. If a chain isn't in state yet, you need one of the above steps first.

Typical first-time flow:

```bash
solver-cli deploy      # deploys contracts, writes .config/state.json
solver-cli configure   # reads state.json, writes .config/hyperlane-relayer.json
```

To add a pre-deployed chain without re-deploying:

```bash
solver-cli chain add --name arbitrum --rpc <url> \
  --input-settler <addr> --output-settler <addr> --oracle <addr> \
  --token USDC=<addr>
solver-cli configure   # regenerates relayer config with the new chain included
```

If Celestia is part of the setup, addresses are also read from `.config/hyperlane-addresses.json`.

## Environment variables

| Variable | Used for |
|---|---|
| `SOLVER_PRIVATE_KEY` | EVM signer key for the relayer |
| `CELESTIA_SIGNER_KEY` | Cosmos signer key (Celestia chain only) |

## Output

`.config/hyperlane-relayer.json` contains:

- One entry per chain with `chainId`, `domainId`, RPC URLs, signer key, and Hyperlane contract addresses (`mailbox`, `merkleTreeHook`, `validatorAnnounce`, IGP)
- A `relayChains` field listing all chains the relayer will relay between

Fields that cannot be resolved from state are written as `PLACEHOLDER` — replace them before starting the relayer.

> **Domain ID note:** chain ID `31337` (local anvil) is mapped to domain `131337` to avoid conflicts with Hyperlane's built-in domain registry.

## Run the relayer

The generated config is volume-mounted into the `relayer` Docker service:

```bash
docker compose up relayer
```

Or directly with the Hyperlane agent binary:

```bash
hyperlane-agent relayer --config .config/hyperlane-relayer.json
```

## Adding a new chain

After adding a chain to `.env` and running `make deploy`, re-run `solver-cli configure` to regenerate the config with the new chain included. Then restart the relayer.
