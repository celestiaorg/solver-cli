# Oracle Operator Service

Independent oracle operator service that watches chains for cross-chain fills and signs attestations.

## Architecture

The oracle operator is a **separate service** from the solver, providing independent verification of fills:

```
┌──────────────┐         ┌──────────────────┐         ┌──────────────┐
│   Solver     │         │ Oracle Operator  │         │ Chains       │
│   Process    │         │    Service       │         │ (Evolve +    │
│              │         │   (this service) │         │  Sepolia)    │
└──────────────┘         └──────────────────┘         └──────────────┘
       │                          │                           │
       │ 1. Fill order            │                           │
       │──────────────────────────────────────────────────────>│
       │                          │                           │
       │                          │ 2. Watch OutputFilled     │
       │                          │<──────────────────────────│
       │                          │                           │
       │                          │ 3. Verify fill happened   │
       │                          │                           │
       │                          │ 4. Sign attestation       │
       │                          │    (operator key)         │
       │                          │                           │
       │                          │ 5. Submit to oracle       │
       │                          │──────────────────────────>│
       │                          │                           │
       │ 6. Poll oracle.isProven()│                           │
       │<──────────────────────────────────────────────────────│
       │                          │                           │
       │ 7. Claim escrowed funds  │                           │
       │──────────────────────────────────────────────────────>│
```

## Why Separate?

**Security**: The solver cannot attest to its own fills. That would be "trust me, I did the work" with no verification. An independent oracle operator provides trust:

- **Solver**: Fills orders (incentivized by profit)
- **Oracle Operator**: Verifies fills (incentivized by service agreement)

## Configuration

The oracle operator uses a separate config file: `config/oracle.toml`

Generated by `solver-cli configure`:

```toml
operator_private_key = "0x..."  # Different key from solver
operator_address = "0x..."      # Must match CentralizedOracle operator

[chains.1234]
chain_id = 1234
rpc_url = "http://127.0.0.1:8545"
oracle_address = "0x..."
output_settler_address = "0x..."

[chains.11155111]
chain_id = 11155111
rpc_url = "https://sepolia..."
oracle_address = "0x..."
output_settler_address = "0x..."
```

## Running

```bash
# Start the oracle operator service
make oracle

# Or manually:
cd oracle-operator
RUST_LOG=info cargo run --release
```

The service will:
1. Connect to both chains
2. Watch for `OutputFilled` events
3. Sign attestations for each fill
4. Submit to `CentralizedOracle` contracts
5. Track processed fills to avoid duplicates

## Environment Variables

- `ORACLE_CONFIG`: Path to config file (default: `config/oracle.toml`)
- `ORACLE_OPERATOR_PK`: Operator private key (set in `.env`)
- `RUST_LOG`: Logging level (e.g., `info`, `debug`)

## Production Considerations

For production use:

1. **Separate Operator Key**: Use a different key from the solver
2. **Monitoring**: Add health checks and metrics
3. **Persistence**: Store processed fills in database
4. **Multi-sig**: Use multi-sig for operator role
5. **Gas Management**: Ensure operator has enough ETH on all chains
6. **Error Recovery**: Handle RPC failures gracefully
7. **Block Reorganizations**: Handle chain reorgs properly

## Testing Locally

For local E2E testing, you can use the same key for both solver and operator (simpler setup), but understand this defeats the trust model and is **only for testing**.
